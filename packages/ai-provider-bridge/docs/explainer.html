What is it?

ai-provider-bridge lets any Posit Assistant platform -- Positron, RStudio, Desktop, TUI -- talk to any LLM provider through a single, shared interface, without platform-specific code leaking into provider logic.

Concretely, it provides:

  • A provider registry (plugin system for registering LLM backends)
  • Model client implementations for 14 providers (Anthropic, OpenAI, Bedrock, etc.)
  • Model discovery with caching (list available models per provider, with TTL and fallbacks)
  • Credential abstractions (platform provides credentials; provider code never knows how they were obtained)
  • An adapter for Positron's vscode.lm API (so extensions using the VS Code Language Model API route through the same infrastructure)
vscode.lm
Positron's VS Code API namespace that lets any extension request LLM completions from models registered by other extensions -- the standard way third-party extensions access AI in Positron.
AI SDK
Vercel's ai npm package -- provides normalized types for messages (ModelMessage) and streaming responses (TextStreamPart). Used as the common wire format between this package and its consumers.

Why does it exist?

The problem

Posit Assistant runs on multiple platforms (Positron, standalone Node, RStudio Workbench, TUI). Each has its own auth system, runtime constraints, and way of discovering models. Before this package, provider logic was entangled with platform-specific code -- making it hard to reuse, test, or ship to new surfaces.

Before

  • Model clients scattered across platform-specific packages
  • Provider registration coupled to platform auth
  • Credential resolution deeply tied to each platform
  • Adding a new consumer meant duplicating provider wiring

After (this package)

  • All provider logic in one standalone package
  • Platform boundary is explicit: only /positron entrypoint touches VS Code
  • Consumers program against interfaces; platforms provide implementations
  • New consumer = just import the package

The key architectural constraint: consumer packages depend on ai-provider-bridge, never the reverse.

Entrypoints

The package exposes four entrypoints, each with a different dependency profile. This is the practical answer to "what do I import?":

Import path What it provides Dependencies Used by
ai-provider-bridge ProviderRegistry, interfaces, types, cached model fetcher, provider map, LocalProviderManager None (platform-neutral) Everyone
ai-provider-bridge/providers register*Provider() functions, all client classes AI SDK provider packages Host applications (Positron, Node platforms)
ai-provider-bridge/providers-external Minimal provider set (Posit AI only, for OSS/external builds) Minimal External/open-source builds
ai-provider-bridge/positron PositronCredentialProvider, VscodeLmClient, message conversion utilities vscode positron-assistant extension only

The root entrypoint is intentionally lightweight -- it defines shapes and the registry, but carries no heavy AI SDK dependencies. Heavy provider code is isolated behind /providers. Only the /positron entrypoint may import vscode.

Key Interfaces

These are the core abstractions that consumers interact with:

Interface ModelClient

The contract every LLM client implements. One method: chat(). Accepts ModelMessage[] (the AI SDK's normalized message format representing user turns, assistant turns, and tool results) and returns a streaming response. 

interface ModelClient {
  chat(params: {
    model: string;
    messages: ModelMessage[];
    systemPrompt?: string;
    tools?: Record<string, AiToolWithJsonSchema>;
    cancellationToken: CancellationToken;
    // ... additional optional params (thinkingEffort, webSearch, etc.)
  }): Promise<AsyncIterable<LMStreamPart>>;
}

// Consuming a stream:
for await (const part of stream) {
  switch (part.type) {
    case "text-delta":    // incremental text chunk
    case "tool-call":     // model wants to call a tool
    case "reasoning":     // extended thinking content
    case "finish-step":   // step complete, includes token usage
  }
}

Every provider -- whether it calls a REST API (Anthropic), spawns a subprocess (Copilot), or routes through VS Code's LM service -- looks identical to callers through this interface.

Class ProviderRegistry

The plugin system. Created once at startup. Providers register a model fetcher (how to list available models) and a client factory (how to create a ModelClient). Consumers ask the registry for models/clients by provider ID -- no hard-coded provider logic in consuming code. 

const registry = new ProviderRegistry(logger);
registerAnthropicProvider(registry, logger);
registerOpenAIProvider(registry, logger);

// Discover models
const models = await registry.getModelsForProvider("anthropic", credentials);

// Create a client and stream a response
const client = registry.getClientForProvider("anthropic", credentials);
const stream = await client.chat({ model: "claude-sonnet-4-5-20250929", messages, ... });
Interface CredentialProvider

Platform-agnostic credential access. The Positron implementation wraps vscode.authentication.getSession(); Node implementations read from env vars, config files, or OAuth flows. Provider code never knows which. 

interface CredentialProvider {
  getCredentials(providerId: ProviderId): Promise<ProviderCredentials | null>;
  onDidChangeCredentials(cb: (ids: ProviderId[]) => void): Disposable;
}

// ProviderCredentials is a discriminated union:
//   ApiKeyCredentials   | OAuthCredentials | LocalCredentials
//   AwsCredentials      | GoogleCloudCredentials
Utility createCachedModelFetcher

Factory for building model fetchers with TTL-based caching and a three-level fallback strategy:

  1. Fresh API fetch -- call the provider's model-listing endpoint
  2. Stale cache -- if the fetch fails, return previously cached results
  3. Static fallback models -- if no cache exists, return a hardcoded list of known models

Most providers use this internally, so model discovery is fast (cache hit) and resilient (never returns nothing).

Key points:

  • Consumers only talk to the ProviderRegistry -- they never reference specific client classes directly.
  • The VscodeLmClient is special: it routes through Positron's LM Service rather than calling a provider API directly. This is how Copilot models (exposed via vscode.lm) get consumed through the same ModelClient.chat() interface.
  • Other VS Code extensions don't import this package at all -- they use the standard vscode.lm API, which the Positron Assistant extension backs with this infrastructure.

Positron-specific components:

  • PositronCredentialProvider -- Implements the CredentialProvider interface for Positron. Uses a static PROVIDER_MAP (a lookup table mapping each provider ID to its Positron auth extension ID and credential type) to translate between logical provider IDs (e.g., "anthropic") and VS Code auth sessions. Handles OAuth tokens, API keys, and AWS credentials uniformly.
  • VscodeLmClient -- A ModelClient implementation that wraps vscode.lm.sendRequest(). Any model exposed through Positron's LM Service (including Copilot) can be consumed through the standard ModelClient.chat() interface.
  • listVscodeLmModels() -- Discovers models available through vscode.lm and enriches them with capability metadata (token limits, tool support, thinking effort levels) using provider-specific heuristics.
  • fromAiMessages2() -- Converts AI SDK ModelMessage[] to VS Code's LanguageModelChatMessage2[] format, handling cache markers, image parts, and tool results in the translation.

Each provider consists of:

  • A client class in src/model-clients/ implementing ModelClient (some providers reuse OpenAIClient with a custom fetch -- e.g., OpenAI-Compatible and MS Foundry)
  • A registration module in src/providers/ exporting register*Provider(registry, logger)
  • An entry in PROVIDER_MAP if it has Positron auth mapping (10 of 14 do; Ollama, LM Studio, OpenRouter, and Google Vertex do not)

Adding a New Provider

  1. Add the provider ID to PROVIDER_IDS in src/types.ts
  2. Create a client class in src/model-clients/MyClient.ts implementing ModelClient
  3. Create a registration module in src/providers/my-provider.ts that exports registerMyProvider(registry, logger)
  4. Export both from src/providers.ts
  5. If it needs Positron auth mapping, add to PROVIDER_MAP in src/provider-map.ts
  6. Register it in your host application's provider setup

The provider inherits caching (via createCachedModelFetcher), error handling, and credential resolution automatically -- typically 50-100 lines of registration code per provider.
